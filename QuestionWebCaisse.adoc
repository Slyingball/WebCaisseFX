# QuestionWebCaisse

## Question 1 — Influence de la réglementation NF525 sur WebCaisse

La norme **NF525** impose que les logiciels de caisse garantissent **l'inaltérabilité, la sécurisation, la conservation et la traçabilité** des données d'encaissement.

Concrètement, pour WebCaisse cela signifie :
- Les tickets de caisse ne peuvent **ni être modifiés ni supprimés** une fois enregistrés.
- Chaque opération doit être **horodatée et tracée** (qui a fait quoi, quand).
- Les données doivent être **conservées 6 ans minimum**.
- Le logiciel doit permettre des **clôtures** journalières, mensuelles et annuelles.

> **Conséquence technique :** 
> Cela impose de concevoir la base de données avec des **tables d'audit** et des mécanismes qui empêchent toute suppression ou modification des données de caisse.

## Question 2 — Envoi d'identifiants par email en clair

C'est une faille de sécurité majeure pour plusieurs raisons :

- **Confidentialité violée** : les emails transitent souvent en clair (protocole SMTP). N'importe qui interceptant le réseau peut lire le mot de passe.
- **Violation du RGPD** : l'article 32 impose des mesures de sécurité adaptées. Envoyer un mot de passe en clair ne respecte pas ce principe.
- **Risque d'usurpation** : si quelqu'un accède à la boîte mail, il récupère directement les identifiants pour se connecter à WebCaisse.

> **Solution sécurisée :** 
> Envoyer un lien à usage unique (token temporaire) par email. Le client clique dessus et **choisit lui-même son mot de passe** via un formulaire sécurisé (HTTPS). Le mot de passe est ensuite **haché** (ex: algorithme bcrypt) avant d'être stocké en base de données.

## Question 3 — Structure de `FormuleSouscrite` inadaptée à l'historique

La table proposée initialement était : `FormuleSouscrite(idPointDeVente, idFormule)` avec une clé primaire composée `(idPointDeVente, idFormule)`.

**Problèmes majeurs** :
- **Pas de notion de temps** : impossible de savoir quand la formule a été activée ou arrêtée → le calcul au prorata pour la facturation mensuelle est impossible.
- **Clé primaire bloquante** : un point de vente ne peut avoir qu'une seule fois la même formule. S'il passe de Gold → Silver → Gold, la base refusera d'insérer le deuxième "Gold" car la combinaison de clés existe déjà.

> **Correction :** 
> Ajouter un identifiant simple `id` en clé primaire (auto-incrémenté) + une `dateDebut` et une `dateFin` (qui vaut `NULL` tant que la formule est active). Chaque changement crée une nouvelle ligne avec sa propre période.

## Question 4 — Intérêt d'un Trigger pour la traçabilité

Un **Trigger** (déclencheur) est un mécanisme SQL qui s'exécute **automatiquement** lors d'un `INSERT`, `UPDATE` ou `DELETE` sur une table.

Ses avantages techniques :
- **Exécution automatique** : pas besoin de rajouter du code Java répétitif. Dès qu'une formule change, le trigger se déclenche en base.
- **Incontournable (Sécurité absolue)** : même si un administrateur modifie la base directement (via phpMyAdmin), la modification est tracée.
- **Indépendance** : la logique de traçabilité est attachée à la base de données, elle continue de fonctionner même si l'application Java est remplacée ou mise à jour.

> **Résultat :** 
> Le trigger enregistre dans une table d'audit dédiée : l'ancienne formule, la nouvelle formule, la date exacte du changement et le point de vente concerné.

## Question 5 — Différence entre `WHERE` et `HAVING`

| Fonctionnalité | Clause `WHERE` | Clause `HAVING` |
| --- | --- | --- |
| **Moment du filtrage** | Filtre les lignes **avant** le regroupement (`GROUP BY`) | Filtre les groupes **après** le regroupement (`GROUP BY`) |
| **Fonctions d'agrégation** | Ne peut **pas** utiliser `COUNT`, `SUM`, `AVG`, etc. | **Peut** et doit utiliser les fonctions d'agrégation |

> **Application dans WebCaisse :** 
> La clause `WHERE` est utilisée en premier pour filtrer les ventes sur une plage de dates et exclure les clients déjà fidèles. 
> Ensuite, la clause `HAVING COUNT(v.id) > seuil` interviendra à la fin, pour ne conserver que les clients dont le **nombre total de ventes** calculé par groupe dépasse le seuil défini.

## Question 6 — Injection SQL par concaténation

L'injection SQL survient lorsque des entrées utilisateur malveillantes sont directement concaténées pour former une requête SQL.

Dans le code vulnérable :
```java
String requete = "... HAVING COUNT(*) > " + seuilVentes;
```

Si un utilisateur saisit le texte malicieux `0; DROP TABLE Conso; --` à la place d'un chiffre, la requête finale devient :
```sql
SELECT ... HAVING COUNT(*) > 0; DROP TABLE Conso; --
```

> **Conséquence destructive :**
> - Le `;` marque la fin de la première requête légitime.
> - Le `DROP TABLE` s'exécute et **supprime instantanément** la table entière de la base.
> - Le `--` indique que la suite (code d'origine) doit être ignorée en tant que commentaire.
> C'est une attaque A03:2021 (Injection) majeure selon l'OWASP.

## Question 7 — Avantages de `PreparedStatement` vs `Statement`

L'utilisation de `PreparedStatement` est standard pour sécuriser les accès de données en Java JDBC.

| Critère | `Statement` classique | `PreparedStatement` |
| --- | --- | --- |
| **Sécurité** | Concaténation = **Vulnérable aux injections SQL** | Utilise des marqueurs `?`. Les données sont "échappées" par le pilote = **Immunisé à l'injection**. |
| **Performance** | Requête analysée et compilée **à chaque exécution**. | Requête **pré-compilée** côté SGBD. Beaucoup plus rapide si elle est exécutée dans une boucle. |
| **Typage** | Tout est traité sous forme de chaîne (`String`). | Les types sont stricts et respectés (`setInt`, `setDate`). |

## Question 8 — `instanceof` dans `compareLieuVente` : intérêt et critique

**L'intérêt** : 
Dans le code actuel, `instanceof` permet de vérifier "à chaud" le type de l'objet manipulé (savoir si une `Vente` est spécifiquement une `VenteEcommerce` ou une `VenteMagasin`). Cela permet de diviser et cumuler correctement les montants dans les paniers e-commerce ou magasin.

**La critique (Anti-pattern POO)** : 
L'usage de nombreux `instanceof` et de `if / else if` indique souvent un défaut de modélisation orientée objet :
- **Violation du principe Ouvert/Fermé (SOLID)** : Si un troisième type de vente apparaît demain (ex: `VenteApplicationMobile`), il faudra retourner modifier à la main la méthode `compareLieuVente` pour ajouter un nouveau `else if`.
- **Manque de polymorphisme** : Il serait préférable que la classe parente abstraite `Vente` ait une méthode `getLieuVente()` retournant une énumération (`MAGASIN`, `ECOMMERCE`), chaque classe fille redéfinissant cette valeur implicitement.

## Question 9 — Division par zéro dans l'indice de comparaison

L’instruction de comparaison finale dans le code est :
```java
return totalMag / totalEcom;
```

> **Le danger systémique :** 
> Si un client fidèle a uniquement effectué des achats en magasin (zéro achat en ligne), la variable `totalEcom` vaudra `0`. 
> En Java, diviser un type primitif `double` par `0.0` ne fait pas planter l'application avec une exception, mais renvoie la métadonnée mathématique absurde **`Infinity`** ou **`NaN`** (Not a Number).

Conséquence : Ce `NaN` va se propager silencieusement dans tout le reste de l'application, détruisant l'affichage et ruinant tous les calculs statistiques et financiers liés à ce client. La correction consiste à placer un simple bloc conditionnel `if(totalEcom == 0)` avant toute division.

## Question 10 — Rôle de `Launcher` et `Main` en JavaFX

Le framework JavaFX a été détaché du JDK officiel à partir de Java 11, et est devenu un système de "modules" strict (JPMS).

- **La classe `Main`** : C'est le vrai programme. Elle contient `extends Application`, configure la scène (`Scene`), charge le fichier XML d'interface (`.fxml`), puis l'affiche. Sa nature modulée provoque aujourd'hui des erreurs au démarrage de la JVM classique (`Error: JavaFX runtime components are missing`).
- **La classe `Launcher`** : C'est une astuce architecturale. C'est une classe basique qui **n'hérite de rien**. Elle contient simplement un `public static void main` dont le seul but est d'appeler l'amorçage de la classe `Main`. 

> **Objectif visé :** 
> Le système de modules de la JVM va inspecter en premier la classe `Launcher`. N'y voyant rien de spécial, elle démarre sans contrainte. `Launcher` charge la vraie classe à la volée. C'est le pattern standard actuel pour faire démarrer JavaFX via Maven sans fichier complexe `module-info.java`.
