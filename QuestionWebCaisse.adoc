## Question 1 — Influence de la réglementation NF525 sur WebCaisse

La norme NF525 impose que les logiciels de caisse garantissent **l'inaltérabilité, la sécurisation, la conservation et la traçabilité** des données d'encaissement.

Concrètement, pour WebCaisse cela signifie :

- Les tickets de caisse ne peuvent **ni être modifiés ni supprimés** une fois enregistrés.
- Chaque opération doit être **horodatée et tracée** (qui a fait quoi, quand).
- Les données doivent être **conservées 6 ans minimum**.
- Le logiciel doit permettre des **clôtures** journalières/mensuelles/annuelles.

/ Cela impose de concevoir la base de données avec des **tables d'audit** et des mécanismes qui empêchent toute suppression ou modification des données de caisse.

## Question 2 — Envoi d'identifiants par email en clair

C'est une faille critique car :

- **Confidentialité violée** : les emails transitent souvent en clair (protocole SMTP). N'importe qui interceptant le réseau peut lire le mot de passe.
- **Violation du RGPD** : l'article 32 impose des mesures de sécurité adaptées. Envoyer un mot de passe en clair ne respecte pas ce principe.
- **Risque d'usurpation** : si quelqu'un accède à la boîte mail, il récupère directement les identifiants.

 **Solution** : envoyer un lien à usage unique (token temporaire) par email. Le client clique dessus et **choisit lui-même son mot de passe** via un formulaire sécurisé (HTTPS). Le mot de passe est ensuite **haché** (bcrypt) avant stockage en base.

## Question 3 — Structure de `FormuleSouscrite` inadaptée à l'historique

La table proposée : `FormuleSouscrite(idPointDeVente, idFormule)` avec clé primaire `(idPointDeVente, idFormule)`.

**Problèmes** :

- **Pas de date** : impossible de savoir quand la formule a été souscrite ou changée → pas de facturation mensuelle possible.
- **Clé primaire bloquante** : un point de vente ne peut avoir qu'une seule fois la même formule. S'il passe de Gold → Silver → Gold, le 2ᵉ Gold est impossible car le couple existe déjà.

 **Correction** : ajouter un `id` auto-incrémenté comme clé primaire + une `dateDebut` et une `dateFin` (NULL si active). Chaque changement crée un nouvel enregistrement avec sa période.

## Question 4 — Intérêt d'un Trigger pour la traçabilité

Un **Trigger** est un mécanisme SQL qui s'exécute **automatiquement** lors d'un INSERT, UPDATE ou DELETE.

Ses avantages :

- **Automatique** : pas besoin que le développeur pense à écrire du code pour tracer — le trigger se déclenche tout seul.
- **Non contournable** : même un accès direct à la base (phpMyAdmin) est tracé.
- **Indépendant de l'application** : la traçabilité fonctionne même si l'application Java est modifiée.

 Le trigger enregistre dans une table d'audit : l'ancienne formule, la nouvelle formule, la date du changement et le point de vente concerné.

## Question 5 — Différence entre `WHERE` et `HAVING`

| `WHERE` | `HAVING` |
| --- | --- |
| Filtre les lignes **avant** le `GROUP BY` | Filtre les groupes **après** le `GROUP BY` |
| Ne peut **pas** utiliser `COUNT`, `SUM`, etc. | **Peut** utiliser les fonctions d'agrégation |

 Dans le cas WebCaisse : `WHERE` filtre les ventes sur une période et exclut les déjà fidèles. `HAVING COUNT(*) > seuil` ne garde ensuite que les consommateurs ayant dépassé le nombre minimum de ventes. Mettre `COUNT(*)` dans le `WHERE` provoquerait une erreur SQL.

## Question 6 — Injection SQL par concaténation

Le code vulnérable concatène la saisie utilisateur directement dans le SQL :
`"... having count(*) > " + seuilVentes`

Si l'utilisateur saisit `0; DROP TABLE Conso; --`, la requête devient :
`SELECT ... HAVING COUNT(*) > 0; DROP TABLE Conso; --`

→ La table `Conso` est **supprimée**.

Le `;` termine la première requête, `DROP TABLE` en injecte une deuxième, et `--` commente le reste. C'est la faille **A03:2021 - Injection** du Top 10 OWASP.

## Question 7 — Avantages de `PreparedStatement` vs `Statement`

| `Statement` | `PreparedStatement` |
| --- | --- |
| Requête par concaténation → vulnérable à l'injection SQL | Marqueurs `?` → **immunisé** contre l'injection |
| Compilée à chaque exécution | **Pré-compilée** → meilleures performances |
| Pas de vérification de type | Types vérifiés (`setInt`, `setString`) |

`PreparedStatement` sépare la **structure SQL** des **données utilisateur**. Le SGBD traite les paramètres comme des valeurs et non comme du code SQL, ce qui empêche toute injection.

## Question 8 — `instanceof` dans `compareLieuVente` : intérêt et critique

**Intérêt** : `instanceof` permet de savoir si une vente est de type `VenteEcommerce` ou `VenteMagasin` pour cumuler les montants dans le bon total.

**Critique** : c'est un **anti-pattern** en POO :

- **Violation du principe Ouvert/Fermé** : ajouter un nouveau type de vente (ex. `VenteMarche`) oblige à modifier la méthode.
- **Violation du polymorphisme** : on devrait plutôt définir une méthode `getLieuVente()` dans `Vente`, redéfinie dans chaque sous-classe, pour ne pas avoir besoin de tester le type.

 Il vaudrait mieux utiliser le **polymorphisme** avec une méthode abstraite plutôt que `instanceof`.

## Question 9 — Division par zéro dans l'indice de comparaison

L'instruction `return totalMag / totalEcom;` est dangereuse car :

- Si `totalEcom == 0` (aucune vente en ligne), la division donne `Infinity` ou `NaN` en Java (avec des `double`), ce qui provoque des **comportements imprévisibles** (affichage erroné, calculs faussés en cascade).

 **Correction** : vérifier que `totalEcom != 0` avant de diviser. Si c'est 0, retourner une valeur par défaut ou lever une exception métier explicite.

## Question 10 — Rôle de `Launcher` et `Main` en JavaFX

Depuis Java 11, JavaFX est un **module externe**. Si la classe principale étend `Application`, la JVM tente de charger JavaFX **avant** le `main` et échoue sans `module-info.java`.

- **`Main`** : la vraie classe JavaFX qui étend `Application`, charge le FXML et affiche la fenêtre.
- **`Launcher`** : une classe simple qui **n'étend pas** `Application`. Elle appelle juste `Main.main()`. Comme elle n'a pas de lien direct avec JavaFX, la JVM ne bloque pas au démarrage.

 `Launcher` sert de **point d'entrée indirect** pour contourner le problème de modules JavaFX sans créer de `module-info.java`. C'est un pattern standard dans les projets JavaFX avec Maven/Gradle.
